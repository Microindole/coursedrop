# 鸿蒙客户端补全计划

## 产品边界

CourseDrop 客户端不做另一个系统文件传输工具。

它要解决的是：

```text
本地优先、端到端加密、支持局域网直传和公网限时中转的跨设备文件投递与分享管理
```

客户端的核心体验不是“选文件发送”，而是管理一次文件投递的完整生命周期：

```text
本地入库 -> 选择分享内容 -> 加密/打包 -> 创建公网分享或局域网连接 -> 上传/直传 -> 扫码/登录下载 -> 过期清理 -> 本地记录
```

因此客户端补全时必须围绕“分享行为、传输状态、过期状态、身份和安全状态”设计，而不是模拟系统目录树。

## 当前已具备

- 设置页可配置多个公网中转源，并可探测健康状态。
- 本机身份可注册到中转服务器，身份信息已持久化。
- 本地库已接入应用专属目录和 RDB/SQLite 索引基础设施。
- 本地库可生成测试文件、导入系统文件/图片、搜索、类型筛选、分页、详情查看、批量选择、加入分享草稿和删除索引。
- 分享页已围绕 CourseDrop 分享草稿、公网链接和上传任务组织。
- 分享草稿可从本地库加入，也可通过系统文件选择器加入。
- 公网分享创建已接入服务端 `/api/shares`。
- multipart 上传已进入主链路。
- 扫码内容解析已有服务边界，可解析网页登录和分享链接。

## 剩余功能总览

### 1. 文件链路稳定化

目标：保证模拟器和真实设备上，本地库不会卡死，文件入库和分享草稿可靠。

需要完成：

- 本地库页面交互回归：滚动、按钮、搜索、筛选、分页、详情抽屉、批量操作。
- 文件导入失败时展示明确错误，不吞掉异常。
- 生成测试文件入口保留为调试能力，后续可隐藏到开发设置。
- 文件大小、MIME、显示名、应用内副本路径必须来自真实文件元数据。
- 删除本地库索引时，需要明确是否只删除索引，还是同时删除应用目录副本。
- 本地库列表重启后必须保持一致，RDB 初始化完成后要刷新页面数据。

验收：

- 新装启动后生成测试文件，列表出现真实文件，重启后仍存在。
- 导入一个小文本文件，大小不是 0，详情页能看到应用内路径。
- 搜索、筛选、分页不会影响批量选择结果的正确性。
- 非详情状态下没有透明浮层拦截滚动和按钮。

### 2. 分享创建与上传

目标：从本地库选择内容后，能够创建公网分享并上传真实文件。

需要完成：

- 分享草稿支持移除单项、清空草稿、重复项提示。
- 创建分享前检查中转源、本机身份、草稿内容和网络状态。
- 上传任务状态要可观察：等待、上传中、成功、失败、取消。
- 失败任务支持重试。
- 分享创建成功但上传失败时，页面要明确显示“链接已创建，文件未完整上传”。
- 撤回分享需要调用服务端撤回接口，而不是只改本地状态。
- 续期分享需要调用服务端续期接口，并同步新的过期时间。

验收：

- 本地库选择文件 -> 加入分享草稿 -> 创建链接 -> 上传完成。
- 服务端分享下载页能看到上传文件。
- 上传失败时能看到具体失败原因，并可重试。
- 撤回后浏览器下载链接不可继续下载。

当前实现进展：

- 分享草稿已支持移除未上传单项和清空未创建草稿。
- 分享创建后上传成功会把服务端分享项 ID 写回草稿项，避免重复上传。
- 上传失败任务已支持从任务列表重试。
- 撤回分享已调用服务端 `DELETE /api/shares/{shareId}`，失败时保留错误提示。
- 续期已调用服务端 `POST /api/shares/{shareId}/expiry`。
- App 下载服务已改为 `/api/shares/{code}/items/{itemId}/download`，下载成功后会写入 `incoming/` 并建立本地库索引。
- App 下载请求已携带本机指纹或账号鉴权头，可走受限下载接口，不再只能依赖浏览器扫码登录路径。
- 分享链接复制已接系统剪贴板。
- 分享二维码改为调用服务端 `/api/qr?text=...` 生成 SVG，客户端直接展示。

仍待完成：

- 服务端单项删除接口还没有接到客户端，已上传项暂不允许从草稿中单独删除。

### 3. 二维码、扫码和浏览器下载

目标：让 CourseDrop 的“二维码投递”成为主体验，而不是只显示链接。

需要完成：

- 分享页展示当前分享二维码，内容应为浏览器下载 URL。
- 二维码区域支持刷新、复制链接、打开系统分享可选入口，但默认不使用系统分享作为主流程。
- 扫码入口独立出来，可用于：
  - 扫网页登录二维码并确认登录。
  - 扫分享二维码并打开下载详情。
  - 扫中转源配置二维码。
- 模拟器无真机时，保留“扫码内容调试”入口，但归入开发/诊断区域。
- 扫码结果分发由 `ScanDispatchService` 负责，页面不解析二维码内容。

验收：

- 电脑浏览器打开下载页后显示网页登录二维码。
- 鸿蒙模拟器通过摄像头扫码，手机身份确认网页登录。
- 分享页二维码可被安卓手机或浏览器扫码打开。
- 输入无效二维码内容时显示明确错误。

### 4. App 内下载与入库

目标：CourseDrop App 不只是上传端，也能作为下载端和本地库管理端。

需要完成：

- App 扫到分享链接后进入下载详情页。
- 下载详情页展示文件列表、大小、过期时间、加密状态和下载策略。
- App 身份鉴权下载接口接入。
- 下载文件写入 `incoming/` 或 `library/`，并写入本地索引。
- 下载任务使用同一套 `TransferTaskService` 状态机。
- 下载完成后可在本地库看到来源和分享会话信息。

验收：

- 用 App 扫分享二维码，可以下载到 CourseDrop 应用目录。
- 下载完成后本地库出现新文件。
- 断网或过期链接给出明确失败态。

当前实现进展：

- 新增 `ShareDownloadPage`，可输入分享码读取分享文件列表。
- 扫码调试识别到分享码后，可以跳转到接收分享页。
- App 下载成功后写入 `incoming/`，并保存到本地库索引。
- 设置页新增系统相机入口，扫码入口从“纯文本调试”中拆出为独立服务边界。
- 扫到包含 `#cdkey=` fragment 的分享链接时，App 接收页会保留 fragment，用于后续 E2EE 解密。

仍待完成：

- 当前 SDK 没有通用二维码解码 API，`CameraScanService` 只能打开系统相机并返回拍摄 URI；需要后续接厂商 Scan Kit 或图像解码库，才能从照片/预览帧自动识别二维码文本。

### 5. 端到端加密

目标：服务器只保存密文和必要元数据，不理解文件明文。

需要完成：

- 明确加密协议：
  - 文件内容使用 AES-GCM。
  - 每个分享或文件生成独立 content key。
  - 密钥通过二维码 URL fragment、扫码登录会话或设备间密钥交换传递，不能上传到服务器。
  - 服务端保存 nonce、tag、算法、KDF 参数、明文大小、密文大小、文件 hash 等元数据。
- 上传前在客户端加密，multipart 上传密文。
- 浏览器下载端使用 WebCrypto 解密。
- App 下载端使用客户端 `EncryptionService` 解密。
- 本地库要显示加密状态：未加密、待加密、已加密、解密失败。

当前实现进展：

- 客户端已用 `cryptoFramework` 替换 XOR 占位加密，上传前生成独立 AES-256-GCM content key。
- multipart 上传的是 `.cdenc` 密文文件，服务端只保存算法、nonce/authTag、明文大小和 SHA-256 等元数据。
- content key 不放进 multipart 表单，也不上传给服务端，而是写入分享 URL 的 `#cdkey=` fragment。
- App 接收页会解析 fragment，下载密文后调用 `EncryptionService.decryptFile` 本地解密，再写入 `incoming/` 和本地库索引。

仍待完成：

- 浏览器下载端的 WebCrypto 解密页面还需要继续实现。
- E2EE key fragment 当前是第一版文本格式，后续应改成版本化 JSON + Base64URL，并支持多文件 key map 的完整性校验。

验收：

- 服务端数据库和文件目录看不到明文。
- 浏览器通过带 `#key=` 的分享链接可下载并解密。
- 缺少 key 时下载页只能拿到密文，不能解密。
- App 内下载能校验完整性，错误 key 明确失败。

### 6. 身份、账号和安全设置

目标：保持“手机指纹优先，账号可选”的产品逻辑。

需要完成：

- 设置页展示本机指纹、账号绑定状态、已绑定设备列表。
- 默认流程不需要账号，手机 App 以指纹身份确认扫码和下载。
- 安全设置中可创建账号，把账号和本机指纹绑定。
- 开启账号登录后，电脑浏览器可用账号密码登录。
- 手机端可解除账号绑定、撤销设备、修改密码。
- 敏感操作需要明确风险提示，例如开启账号密码登录会降低“纯扫码”默认安全模型。

验收：

- 未创建账号时，浏览器只能用手机扫码登录。
- 创建账号后，浏览器可账号密码登录，也可扫码登录。
- 一个账号可绑定多个手机指纹。

当前实现进展：

- 设置页已支持创建账号并绑定当前设备指纹。
- 设置页可开启或关闭账号密码登录。

仍待完成：

- 已绑定设备列表、解绑设备、修改密码页面还需继续补。

### 7. 局域网发现与直传

目标：体现和单纯公网中转不同的“本地优先”能力。

需要完成：

- 局域网设备发现服务边界落地，优先考虑 mDNS/局域网广播可行方案。
- 设备页展示真实在线设备、平台、延迟、连接方式。
- 传输策略选择：
  - 同局域网优先直传。
  - 直连失败后提示切换公网中转。
  - 用户可在设置中关闭局域网优先。
- 直传任务和公网上传/下载共用传输任务模型。

当前实现进展：

- 设备发现服务已改为异步接口。
- 设备页点击扫描会调用 `LanDeviceDiscoveryService.refresh()`，不再通过页面 `setTimeout` 模拟扫描。
- `LanDeviceDiscoveryService` 已基于 UDP 广播发送 `coursedrop.lan.v1` 发现报文，并把收到的同协议节点写入 `DeviceRegistry`。

仍待完成：

- 当前只完成发现层，真正 TCP/UDP 文件直传服务、监听端、握手协议和断点/进度回调还未完成。
- 需要至少两台运行 CourseDrop 局域网协议的设备才能验证发现结果；单模拟器环境可能只能看到空状态。

验收：

- 同网络两台设备能互相发现。
- 直传失败不影响公网中转分享。
- 设备页不展示假设备。

### 8. 诊断、日志和测试入口

目标：解决模拟器开发难调试的问题。

需要完成：

- 设置页或诊断页展示：
  - 当前中转源。
  - 本机身份。
  - 最近一次 API 请求结果。
  - 最近一次上传/下载错误。
  - 本地库根目录。
- 所有关键服务统一使用 `[CourseDrop]` 日志标签。
- 诊断页提供：
  - 生成测试文件。
  - 输入扫码内容。
  - 探测中转源。
  - 查看本地索引数量。

验收：

- 出现“没反应”时，可以从页面或 HiLog 快速定位是 UI、文件、网络、身份还是服务端问题。

## 推荐实施顺序

不要一次性并行做所有功能。建议按下面顺序推进：

1. **稳定本地库与分享草稿**
   - 修本地库交互、分页、详情抽屉和批量操作。
   - 草稿支持移除、清空、重复提示。

2. **跑通公网上传下载闭环**
   - 创建分享、上传真实文件、浏览器下载。
   - 补撤回、续期、失败重试。

3. **补二维码与扫码登录**
   - 分享二维码展示。
   - 浏览器网页登录二维码确认。
   - App 扫分享链接进入下载详情。

4. **补 App 内下载入库**
   - App 下载任务。
   - 写入应用目录。
   - 下载后进入本地库。

5. **接端到端加密**
   - 先加密上传和浏览器解密。
   - 再补 App 下载解密。

6. **接账号安全与设备管理**
   - 账号创建、指纹绑定、多设备管理。
   - 安全设置页补全。

7. **接局域网发现和直传**
   - 设备发现。
   - 直传策略。
   - 公网兜底。

## 当前下一步

当前最优先不是局域网，也不是账号系统，而是：

```text
本地库选文件 -> 分享草稿 -> 创建公网链接 -> 上传文件 -> 浏览器下载
```

这个闭环一旦稳定，CourseDrop 就和系统文件传输拉开差异：它不是近场传输，而是可管理、可过期、可登录、可浏览器下载的临时文件投递。
